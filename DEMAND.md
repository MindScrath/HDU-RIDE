# 业务逻辑与正确反馈需求书

## 1. 文档目标

本文档用于定义 HDU RIDE 下一阶段的目标业务模型、权限体系、页面行为与正确反馈规范，作为后续 AI 直接实现的正式需求输入。

本文档不是对当前实现的说明，而是目标需求说明。若本文档与当前代码实现冲突，以本文档为准。

本文档重点解决以下问题：

- 课程在当前系统中不是独立实体，无法支撑真正的课程级管理
- `admin` 当前是全局角色，但目标应为独立身份的课程级管理员
- `assistant` 当前被当作账号角色使用，但目标中助教只是班级标签
- 课程讲义仍是离线导入/重载模式，目标中需要可在线管理
- 后台页面虽有基础 CRUD，但交互体验与反馈远不够统一

## 2. 设计原则

### 2.1 总体原则

- 系统服务于学校教学场景，必须围绕“课程 -> 教学组 -> 班级 -> 学生 -> 作业 -> 批改”展开
- 一切权限都必须有明确归属层级，不能出现同一身份在多个层级语义混乱
- 一切增删改查操作都必须有明确的成功反馈、失败反馈、权限反馈和确认反馈
- 一切页面都必须首先满足正确性，其次再追求高级感和视觉统一
- 面向 AI 实现，文档必须避免“差不多”“自行判断”“看情况处理”这类表达

### 2.2 核心业务原则

- 课程必须是一级业务实体，不再只是一个 `course_id` 字符串
- `root` 是系统唯一最高管理员，系统中不允许出现第二个 root
- `admin` 是独立身份的课程级管理员，不再是单纯全局角色，且不能兼任 `teacher`
- `teacher` 是授课教师，是班级和学生交互的主要角色
- `student` 是学生，是作业完成者
- 助教不是一种账号类型，不是全局角色，只是某个班级中的一个成员标签
- 同一个学生可以在多个班级被设置为助教，且这些班级可以属于不同课程
- 班级至少要有一名授课教师
- 课程内容由课程承载，班级只引用课程，不重复拥有课程内容

### 2.3 实现策略原则

- 为降低改造成本，课程内容本期允许继续以文件方式存储，但业务上必须由课程实体管理
- 讲义编辑器本期按 MVP 文件编辑器实现，不要求一次达到完整 IDE 级别
- 本期不做需求分期，直接定义完整目标态

## 3. 目标业务模型

## 3.1 课程

课程是一级业务实体，代表一个教学组和该教学组下的教学内容。

课程至少包含以下属性：

- `id`
- `name`
- `code`
- `description`
- `status`
- `contentRoot` 或课程内容目录绑定信息
- `createdAt`
- `updatedAt`
- `createdBy`

课程的职责：

- 承载课程级管理员与教师教学组关系
- 承载讲义与作业内容
- 作为班级的上级实体
- 为学生提供课程教案

约束：

- 课程创建时必须至少指定一名课程级 `admin`
- 课程允许拥有多名 `admin`
- 课程允许拥有多名 `teacher`
- 课程删除前必须有明确的风险确认

## 3.2 课程成员

课程成员分为两类：

- 课程管理员 `admin`
- 课程教师 `teacher`

课程成员关系必须独立建模，不允许再通过单一全局角色硬编码推断课程归属。

课程成员关系至少要包含：

- `courseId`
- `userId`
- `memberRole`，取值仅允许 `admin` 或 `teacher`
- `joinedAt`
- `invitedBy`

约束：

- 一个用户可以加入多个课程
- 同一用户不能同时兼任 `admin` 与 `teacher`
- 若用户是 `admin`，则其职责是课程级管理，不承担授课教师职责
- 课程至少保留一名 `admin`

## 3.3 班级

班级是课程下的教学执行单位。

班级至少包含以下属性：

- `id`
- `courseId`
- `name`
- `term`
- `description`
- `status`
- `createdAt`
- `updatedAt`
- `createdBy`

约束：

- 一个班级必须且只能属于一个课程
- 一个班级至少有一名授课教师
- root 可以创建任意班级，但必须指定授课教师
- admin 不能以授课教师身份创建班级
- teacher 只能在自己已加入的课程下创建班级

## 3.4 班级成员

班级成员分为三类：

- 授课教师
- 学生
- 学生助教标签

其中“助教”不是独立账号类型，而是班级成员上的附加标签。

班级成员关系至少应包含：

- `classId`
- `userId`
- `memberType`，取值允许 `teacher` 或 `student`
- `assistantFlag`
- `joinedAt`
- `joinedBy`

约束：

- `assistantFlag=true` 时，`memberType` 必须为 `student`
- 同一用户在同一班级只能有一条成员记录
- 学生可以在班级 A 和班级 B 同时被标记为助教
- 班级至少保留一名教师成员

## 3.5 讲义

讲义属于课程，不属于班级。

讲义内容可继续采用课程目录文件方式存储，但业务上必须满足以下要求：

- 每份讲义都归属于某个课程
- 讲义可被课程内所有教师与 admin 管理
- 课程下所有班级学生可查看该课程讲义
- 班级不复制讲义，只引用课程讲义

讲义至少要支持：

- 章节树
- Markdown 内容
- 排序
- 草稿/已发布状态

## 3.6 作业

作业属于班级，不属于课程。

作业至少包含：

- `id`
- `title`
- `description`
- `openAt`
- `dueAt`
- `rstudioImage`
- `starterPath`
- `submitPath`
- `status`

作业要求：

- 每份作业必须归属于一个明确的班级
- 作业由班级教师创建与维护
- root 可查看和管理任意班级作业
- admin 可查看自己管理课程下各班级的作业与执行情况
- 班级学生只能看到自己所在班级的作业
- 班级教师和助教可批改该班级学生提交

## 3.7 提交与批改

提交与批改沿用“工作区归档 -> 历史复核 -> 成绩发布”的总体思路，但目标模型必须更清晰。

提交至少包含：

- `id`
- `classId`
- `assignmentId`
- `studentId`
- `attempt`
- `textObject`
- `fileObject`
- `late`
- `submittedAt`

成绩至少包含：

- `id`
- `submissionId`
- `score`
- `comment`
- `graderId`
- `publishedAt`
- `updatedAt`

## 4. 角色体系与权限要求

## 4.1 Root

定义：

- 系统唯一最高管理员
- 拥有所有课程、班级、讲义、作业、用户的最高权限

硬性约束：

- 整个系统唯一一个 root
- 不允许任何页面、接口、导入逻辑、批量操作、新建用户流程创建第二个 root
- 非 root 不可修改 root 身份
- 非 root 不可禁用 root

权限：

- 创建、查看、编辑、删除课程
- 指定课程 admin
- 查看和管理所有课程成员
- 查看和管理所有班级
- 查看和管理所有作业、提交、评分
- 查看所有讲义
- 在必要时介入修正课程与班级关系

限制：

- root 不应被视为班级默认成员
- root 不应自动占用班级中的“授课教师”名额

## 4.2 Admin

定义：

- 课程级管理员，通常是教学组组长
- admin 的权限只在所属课程范围内生效

权限：

- 查看自己所管理的课程
- 维护课程基本信息
- 邀请教师加入课程教学组
- 移除课程内教师
- 在课程内维护讲义
- 查看课程下所有班级
- 查看课程下班级成员与作业执行情况
- 查看和批阅课程下所有班级作业

限制：

- admin 不能越权管理不属于自己的课程
- admin 不能创建 root
- admin 不能管理 root
- admin 不能把某个用户提升为全局超级管理员

## 4.3 Teacher

定义：

- 已加入课程教学组的授课教师

权限：

- 查看自己加入课程的讲义与作业
- 在所属课程下创建班级
- 管理自己授课班级的学生
- 在自己授课班级中给学生打助教标签
- 打开并批改自己授课班级的作业提交
- 导出自己授课班级的成绩

限制：

- teacher 不能管理不属于自己的课程
- teacher 不能创建课程
- teacher 不能任意管理其他教师的课程
- teacher 不能把用户提升为课程 admin

## 4.4 Student

定义：

- 学习者

权限：

- 查看自己所在班级所属课程的讲义
- 查看自己所在班级可见的作业
- 打开自己的作业工作区
- 提交自己的作业
- 查看自己的提交状态与成绩

限制：

- student 不可管理课程
- student 不可管理班级
- student 不可查看其他学生提交
- student 不可批改作业

## 4.5 Assistant

定义：

- assistant 不是账号角色
- assistant 是某个班级中的 `student + assistantFlag=true`

权限：

- 查看其所在班级成员
- 协助管理班级学生成员
- 协助批改本班级作业

限制：

- assistant 不能给别人下发更高身份
- assistant 不能管理班级教师
- assistant 不能修改课程信息
- assistant 的权限只在被标记的班级内生效
- assistant 跨班级不自动继承，必须逐班级设置

## 5. 功能需求

## 5.1 Root 功能需求

### 课程

- 课程是一级菜单
- root 可以查看课程列表
- root 可以新建课程
- root 可以编辑课程
- root 可以删除课程
- root 可以查看课程下 admin 与 teacher
- root 可以为课程新增或移除 admin
- root 可以为课程新增或移除 teacher

课程创建页至少应支持：

- 课程名称
- 课程编码
- 课程简介
- 初始课程 admin 选择
- 课程内容目录绑定信息或初始课程内容包

课程详情页至少包含三个区块：

- 课程基本信息
- 教学组成员管理
- 讲义管理入口
- 课程下班级入口

### 班级

- root 可以查看所有班级
- root 可以按课程筛选班级
- root 可以创建任意课程下的班级
- root 创建班级时必须指定至少一名授课教师
- root 可以管理班级中的教师、学生、助教标签

### 讲义

- root 可以查看任意课程讲义
- root 可以进入课程讲义编辑器
- root 可以发布、撤回、删除讲义

### 作业

- root 可以查看、修改、批阅任何课程任何班级的作业
- root 可以查看任何班级提交与评分

## 5.2 Admin 功能需求

### 课程成员管理

- admin 可以查看自己管理课程的成员列表
- admin 可以邀请 teacher 加入课程
- admin 可以移除课程中的 teacher
- admin 不可移除课程中最后一名 admin
- admin 不可把普通 teacher 提升为 root
- admin 不兼任 teacher，不能直接承担授课班级教师职责

课程内邀请 teacher 的导入规则必须明确如下：

- 导入格式优先支持两列：`账号`、`密码`
- 若账号不存在，则创建 `teacher` 账号并加入当前课程
- 若账号已存在，且身份允许作为 `teacher` 加入课程，则只执行“加入课程”，不得重复修改其密码
- 若账号已存在但身份与目标不兼容，则必须明确报错，不得强行覆盖原身份
- “邀请/导入 teacher” 的核心目标是建立课程成员关系，不是批量重置密码

### 讲义管理

- admin 可以创建讲义章节
- admin 可以新建 Markdown 讲义文件
- admin 可以编辑讲义内容
- admin 可以重命名讲义文件
- admin 可以调整讲义顺序
- admin 可以删除讲义文件或章节
- admin 可以预览讲义渲染结果
- admin 可以发布讲义

### 班级管理

- admin 可以查看课程下所有班级
- admin 可以查看班级教师配置是否完整
- admin 可以查看课程下各班级成员情况
- admin 可以查看课程下各班级作业执行情况

## 5.3 Teacher 功能需求

### 班级创建与管理

- teacher 只能在自己加入的课程下创建班级
- teacher 创建班级时默认成为该班级授课教师之一
- teacher 可以为自己班级再增加教师
- teacher 不可把不属于该课程的教师直接加入班级，必须先加入课程教学组

### 学生管理

- teacher 可以导入学生
- teacher 可以移除学生
- teacher 可以在本班级中把学生设为助教
- teacher 可以取消某学生在本班级的助教标签
- teacher 可以重置学生密码

学生导入规则必须明确如下：

- 导入格式优先支持两列：`账号`、`密码`
- 若账号不存在，则创建 `student` 账号并加入当前班级
- 若账号已存在，且身份允许作为 `student` 加入班级，则只执行“加入班级”，不得重复修改其密码
- 若账号已存在于本班级，则导入行为应幂等，不得重复创建成员记录
- 若账号已存在但身份与目标不兼容，则必须明确报错，不得强行覆盖原身份
- “导入学生” 的核心目标是建立班级成员关系，不是批量重置密码

### 作业与批改

- teacher 可以查看本班级学生的作业列表
- teacher 可以创建本班级作业
- teacher 可以编辑本班级作业时间、标题、说明、工作区配置
- teacher 可以查看本班级学生提交历史
- teacher 可以恢复历史提交到工作区进行查看
- teacher 可以评分并发布成绩
- teacher 可以导出班级作业成绩

## 5.4 Student 功能需求

### 课程学习

- student 可以看到自己所在班级所属课程的讲义
- student 可能同时属于多个课程，因此需要先能切换课程或从班级上下文进入

### 作业完成

- student 可以查看自己所在班级的作业
- student 可以打开作业对应的 RStudio 工作区
- student 可以提交当前工作区
- student 可以看到自己的提交次数、最后提交时间、是否晚交、成绩与评语

### 访问限制

- student 不可查看其他学生提交内容
- student 不可进入讲义编辑器
- student 不可管理课程和班级成员

## 5.5 Assistant 功能需求

- assistant 可以查看本班级成员列表
- assistant 可以帮助管理学生成员
- assistant 可以协助批改本班级作业
- assistant 不可新增或删除班级教师
- assistant 不可修改课程配置
- assistant 不可给其他学生下发“更高身份”

## 6. 按业务对象的详细需求

## 6.1 课程菜单

课程菜单应替代当前单纯“课程导入”概念，成为完整课程工作台。

课程菜单至少包含：

- 课程列表
- 新建课程
- 课程详情
- 教学组成员
- 讲义管理
- 课程下班级入口

课程列表字段至少包含：

- 课程名称
- 课程编码
- 课程状态
- admin 数量
- teacher 数量
- 班级数量
- 更新时间

## 6.2 课程讲义编辑器

讲义编辑器按在线 MVP 文件编辑器实现，视觉参考在线 VS Code，但不要求完整 IDE 能力。

必须支持：

- 左侧课程目录树
- 新建文件
- 新建章节目录
- 重命名
- 删除
- Markdown 编辑
- 保存
- 预览
- 章节排序
- 未保存状态提示

本期明确不强制要求：

- 多标签编辑
- 全局搜索替换
- Git 差异对比
- 插件系统

## 6.3 班级菜单

班级菜单需要支持：

- 班级列表
- 创建班级
- 查看班级详情
- 管理班级教师
- 管理学生
- 设置/取消助教标签
- 查看讲义
- 查看作业

班级详情至少包含：

- 基本信息
- 所属课程
- 授课教师
- 学生人数
- 助教人数
- 最近作业情况

## 6.4 作业菜单

作业是班级级对象，必须至少支持两种视角：

- 学生视角：完成作业
- 教师/Admin/Root 视角：查看提交、批改、导出

教师视角必须支持：

- 创建班级作业
- 编辑班级作业
- 查看最新提交
- 查看历史提交
- 进入复核工作区
- 评分
- 发布成绩
- 导出 CSV

学生视角必须支持：

- 查看题面
- 打开工作区
- 提交
- 查看成绩

## 7. 页面与交互需求

## 7.1 总体风格

- 所有增删改查页面采用仿 Notion Database 风格
- 页面风格应统一、简洁、高级、现代
- 列表优先使用高信息密度表格视图
- 操作按钮位置、筛选条、批量操作栏、详情抽屉风格应统一

## 7.2 所有 CRUD 页面统一要求

适用范围：

- 课程
- 课程成员
- 班级
- 班级成员
- 讲义目录
- 作业管理
- 用户管理

统一要求：

- 顶部有标题与简短说明
- 支持搜索
- 支持筛选
- 支持批量操作
- 支持空状态页
- 支持危险操作确认
- 支持行级快捷操作
- 支持详情页或详情抽屉

## 7.3 状态呈现要求

页面中必须明确呈现以下状态：

- 加载中
- 无数据
- 无权限
- 操作成功
- 操作失败
- 未保存变更
- 高风险操作待确认

## 8. 正确反馈规范

## 8.1 成功反馈

所有创建、编辑、删除、发布、导入、邀请、设置助教、提交作业、评分等操作都必须有明确成功反馈。

要求：

- 使用清晰中文提示
- 成功反馈必须与动作对应
- 成功后页面数据应立即刷新或局部更新

示例：

- “课程已创建”
- “课程管理员已添加”
- “教师已加入课程”
- “教师账号已创建并加入课程”
- “班级已创建”
- “学生已导入”
- “学生账号已创建并加入班级”
- “已设为本班助教”
- “讲义已保存”
- “成绩已发布”

## 8.2 失败反馈

失败反馈必须尽量区分失败原因，禁止大量使用模糊提示如“操作失败”而不给原因。

至少需要区分：

- 参数错误
- 权限不足
- 资源不存在
- 状态冲突
- 业务规则不允许
- 系统异常

示例：

- “课程名称不能为空”
- “该课程至少需要保留一名管理员”
- “你无权管理该课程”
- “该账号已存在，本次仅加入课程，密码未修改”
- “该账号已存在，本次仅加入班级，密码未修改”
- “该学生不在当前班级中”
- “班级至少需要一名授课教师”
- “该用户尚未加入课程，不能直接加入班级授课”
- “该账号当前身份与目标身份不兼容，不能直接导入”

## 8.3 权限反馈

权限不足时必须满足：

- 接口返回明确权限错误
- 前端页面不展示不该展示的操作按钮
- 即使前端误展示，后端也必须拦截

前端提示建议：

- “你没有权限执行此操作”
- “你只能管理自己所属课程”
- “你只能批改自己班级的作业”

## 8.4 空状态反馈

以下场景必须有友好的空状态：

- 还没有课程
- 课程内还没有教师
- 课程内还没有讲义
- 班级内还没有学生
- 该作业还没有提交
- 学生还没有任何作业

空状态文案要说明“当前是什么情况”和“下一步建议做什么”。

## 8.5 危险操作确认

以下操作必须二次确认：

- 删除课程
- 删除班级
- 移除课程 admin
- 移除课程 teacher
- 移除班级教师
- 批量删除成员
- 删除讲义文件
- 发布成绩

确认文案必须明确后果，禁止只写“是否确认”。

## 9. 与当前实现的明确差异

以下旧逻辑在目标实现中不应继续沿用：

- 课程不再只是班级上的 `course_id` 字符串，而是独立实体
- `admin` 不再只是全局角色，而是独立身份的课程级成员关系，且不能兼任 `teacher`
- `assistant` 不再是全局账号角色，而只存在于班级成员标签中
- 课程讲义不能只靠 zip 导入 + reload 管理，必须具备系统内编辑能力
- 作业不再归属于课程，而是归属于班级
- 页面不能只提供基础 CRUD，必须补齐统一交互、状态反馈和空状态

以下现有能力可继续保留并升级：

- 作业工作区与归档机制
- 提交历史与批改机制
- 成绩导出
- 班级视角的讲义与作业访问

## 10. 验收标准

在所有业务验收之前，必须先经过测试阶段。对于系统增删改查相关功能接口，只有对应 `TESTbench` 编写完成且执行通过，才允许进入“已达成目标”的判定。

## 10.1 业务模型验收

- 可以创建独立课程实体
- 创建课程时必须指定至少一名课程 admin
- 一个用户可同时管理多个课程
- 一个用户可同时属于多个课程，但不得同时兼任 admin 与 teacher
- 助教只存在于班级内，不再是全局账号类型
- 一个学生可在多个班级分别拥有助教标签
- 作业明确归属于班级，而不归属于课程

## 10.2 权限验收

- 系统中无法通过任何正常路径创建第二个 root
- root 可以管理所有课程与班级
- admin 只能管理所属课程
- admin 不能兼任 teacher，也不能直接行使授课教师创建班级的能力
- teacher 只能在所属课程下创建班级
- assistant 只能在被标记的班级内协助管理
- student 只能查看自己的学习与作业数据

## 10.3 页面验收

- 课程、班级、成员、讲义、作业等管理页都具备统一的 Notion 风格 CRUD 体验
- 讲义编辑器能完成 MVP 要求的目录管理、编辑、保存和预览
- 所有核心列表页都具备搜索、筛选、批量操作和空状态

## 10.4 正确反馈验收

- 所有核心操作都有明确成功提示
- 常见失败场景都有明确错误提示
- 危险操作都有二次确认
- 权限不足时前后端表现一致
- 空数据状态不会显示空白页面

## 10.5 作业闭环验收

- student 可以完成“查看作业 -> 打开工作区 -> 提交 -> 查看成绩”的完整闭环
- teacher 可以完成“查看提交 -> 打开历史版本 -> 评分 -> 发布 -> 导出”的完整闭环
- assistant 可以协助完成其所在班级的成员管理与批改

## 10.6 测试阶段验收

- 必须存在明确的测试阶段，不能只靠人工点点页面来判断完成
- 对系统增删改查相关接口，AI 必须补充对应的 `TESTbench`
- `TESTbench` 必须覆盖成功路径、常见失败路径、权限路径和幂等性/重复提交路径
- 所有 `TESTbench` 执行通过后，才算该接口达到可交付状态
- 若某接口没有对应 `TESTbench`，即使功能表面可用，也不算完成目标
- 若修改了已有接口行为，必须同步更新原有测试或新增测试，不能只改实现不改测试

## 11. 对 AI 实现的要求

后续 AI 实现必须遵循以下原则：

- 先完成目标数据模型和权限模型，再做页面补齐
- 前后端都必须按本文档实现，不允许只改前端不改后端
- 一切旧逻辑若与本文档冲突，必须以新需求重构
- 在成员导入、账号创建、班级成员绑定、作业提交与批改等场景中，优先复用已有实现与已有接口语义，尽量在现有代码基础上演进，不要无必要另起炉灶
- 若现有实现已经覆盖部分流程，应优先保留成熟链路，只修正与本文档冲突的部分，例如“已有账号导入时不要重复修改密码”
- 对所有增删改查相关接口，必须编写并跑通对应 `TESTbench`，测试通过后才算实现完成
- 优先复用当前仓库已有测试组织方式与已有测试辅助逻辑，在现有测试体系上扩展，不要平行再造一套完全脱离当前项目结构的测试框架
- 实现过程中如果出现文档未覆盖但影响核心业务的细节，应优先补全文档或显式记录假设，而不是私自改变核心业务原则
